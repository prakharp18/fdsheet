# Hydration

Hydration is the process where a framework takes a static HTML page generated by the server and makes it interactive by attaching event listeners and state to it.

It acts as the bridge between the "dry" HTML sent by the server and the "wet" interactive React/Vue application required by the browser.

:::info[Key Idea]
Hydration does **not** rebuild the UI. It simply walks the existing DOM tree and "claims" the elements, restoring state and behavior without re-rendering the visual structure.
:::

---

## 1. Internal Working

When a user requests a page, the server generates the full HTML and sends it to the browser. At this point, the page is visible but totally non-interactive.

1. **Server Rendering**: The framework converts components into an HTML string.
2. **Client Booting**: The browser downloads the JavaScript bundle.
3. **The Replay Phase**: The framework runs the application code on the client. It matches existing DOM nodes with the Virtual DOM structure and attaches event handlers (`onClick`, `onChange`).

### The Hydration Flow

```mermaid
sequenceDiagram
    participant S as Server
    participant B as Browser
    participant JS as Client JS Bundle

    S->>B: Sends Static HTML (Content Visible)
    B->>JS: Downloads JS Assets
    Note over B,JS: Page is non-interactive
    JS->>B: Execute Framework Code
    JS->>B: Replay/Match Virtual DOM to HTML
    JS->>B: Attach Event Listeners
    Note over B,JS: Hydration Complete (Interactive)
```

:::warning[Common Mistake]
**Direct DOM Manipulation**: Changing the DOM before hydration starts (like a script removing a div) can cause the framework to lose track of nodes, leading to "Hydration Mismatch" errors.
:::

---

## 2. Real-World Example

Imagine reading a blog post with a "Like" button rendered on the server.

- **Before Hydration**: You see the button instantly, but clicking it does nothing because the `onClick` function hasn't been attached yet.
- **After Hydration**: The JavaScript has "hydrated" the button. Clicking it now updates the state properly.

---

## 3. Why it Matters

1. **SEO**: Search engine crawlers see a fully populated HTML page without needing to execute heavy JS.
2. **Perceived Performance**: Users see content immediately (fast LCP) even if they have to wait a moment for interactivity.

:::tip[Interview Insight]
**Q: How does Hydration differ from CSR (Client Side Rendering)?**

In CSR, the browser starts with an empty `<div id="root"></div>` and builds the DOM entirely from scratch. In Hydration, the browser starts with a full DOM and only adds "intelligence."
:::
